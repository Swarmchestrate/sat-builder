# Swarmchestrate Template Builder Configuration

This repository contains configuration files used by the Swarmchestrate Application Template Builder API. These files define the API metadata, supported template types, routing configuration, and default TOSCA template structures for generating Application and Capacity templates.

## Files Overview

The three main configuration files are:

1. `app.yaml`: Application-related configuration.
2. `tosca_application_template.yaml`: Application endpoint schema constraints and configuration.
3. `tosca_capacity_template.yaml`: Capacity endpoint schema constraints and configuration.

> [!WARNING]
> Changes to these files may break the API, generated templates, or client integrations.
> Make sure you understand the impact before modifying them.

---

## Relationship Between the Files

The three files work together as follows:

### `app.yaml`

Acts as the main API configuration file.

It tells the API which template types are available and which configuration files correspond to them.

### `tosca_application_template.yaml`

Defines the TOSCA Application Template structure and validation rules.

It is used to generate the corresponding schemas when clients call the Application Template build endpoint.

### `tosca_capacity_template.yaml`

Defines the TOSCA Capacity Template structure and validation rules.

It is used to generate the corresponding schemas when clients call the Capacity Template build endpoint.

---

## Description of the Three Main Configuration Files

### `app.yaml`

The `app.yaml` file defines the main API configuration for the Swarmchestrate Application Template Builder.

It includes:

- API metadata, such as title, version, license, and contact information.
- Documentation endpoints, including Swagger UI, ReDoc, and OpenAPI schema paths.
- Swagger UI settings, such as filtering, request duration display, and try-it-out support.
- Supported TOSCA template types and their corresponding configuration files and template directories.
- OpenAPI tags used to group endpoints in the documentation.
- Router configuration for:
  - `/`
  - `/health`
  - `/application/build`
  - `/capacity/build`
- Validation settings, including enabling or disabling TOSCA validation with the [Sardou TOSCA Toolkit](https://pypi.org/project/Sardou/).

### `tosca_application_template.yaml`

The `tosca_application_template.yaml` file defines the default TOSCA structure and schema constraints for generating Swarmchestrate Application Templates.

It is used by the API to build TOSCA-based application templates and to dynamically generate the related API schema constraints.

It includes:

- The default TOSCA definitions version.
- Default imports and metadata.
- The default request body structure through the service template.
- Node schema constraints for validating node fields.
- Policy constraints, such as valid ranges for `priority` and `target`.
- Default API parameters, including response type and template version.

Changes to this file are reflected in the generated API schema constraints.

### `tosca_capacity_template.yaml`

The `tosca_capacity_template.yaml` file defines the default TOSCA structure and schema constraints for generating Swarmchestrate Capacity Templates.

It is used by the API to build TOSCA-based capacity templates and to dynamically generate the related API schema constraints.

It includes:

- The default TOSCA definitions version.
- Default imports and metadata.
- The default request body structure through the service template.
- Node schema constraints for validating capacity node fields.
- Policy constraints, such as valid ranges for `priority` and `target`.
- Default API parameters, including response type and template version.

Changes to this file are reflected in the generated API schema constraints.

## Generated Configuration Files

When the API starts, the service dynamically generates additional configuration files based on the two TOSCA configuration files listed above.

Specifically, the service generates:

- `openapi.json` files that are loaded into the API schema and documentation.
- `schema.json` files that define the constraints used to validate requests.

These files are created for both the Application and Capacity endpoints, and for both TOSCA node templates and policies.

After successful initialization, the following files are created:

- `tosca_application_template_node_openapi.json`
- `tosca_capacity_template_node_openapi.json`
- `tosca_application_template_node_schema.json`
- `tosca_capacity_template_node_schema.json`
- `tosca_application_template_policy_openapi.json`
- `tosca_capacity_template_policy_openapi.json`
- `tosca_application_template_policy_schema.json`
- `tosca_capacity_template_policy_schema.json`

> [!WARNING]
> These files are generated by the service during initialization.
> Manual changes may be overwritten or may cause schema and validation issues.

## Available Template Types

The API supports generating two template types:

- `application`
- `capacity`

Both template types allow fields based on the configured TOSCA node templates and policies.

### Application Templates

Application Templates describe deployable software components and their requirements.

They may include:

- Application containers
- Environment variables
- Ports
- Arguments
- Host requirements
- Volume requirements
- Resource selection constraints
- QoS policies

### Capacity Templates

Capacity Templates describe available infrastructure resources.

They may include:

- Cloud resources
- Resource types
- Capacity provider information
- Energy policies
- Resource-level validation rules

## Default Response Behavior

Both template configuration files define the same default response behavior:

This means the API returns generated templates in both YAML and JSON formats by default and uses the latest template version unless another version is requested.

The API can also generate templates only in YAML or JSON formats by changing the correspoding configuration.

- The Application and Capacity template files provide reusable defaults for the API while allowing clients to generate customized templates through API requests.
